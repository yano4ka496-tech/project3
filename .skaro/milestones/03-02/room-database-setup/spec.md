# Specification: room-database-setup

## Context
Настройка Room базы данных с шифрованием для SafePlant. Задача решает проблему безопасного хранения чувствительных данных (допусков, результатов квиза, сопоставлений QR-кодов) в офлайн-режиме. Шифрование реализуется через **SQLCipher** (шифрование файла БД) с ключом, защищённым **Android Keystore**. Требуется создание сущностей, DAO, миграций и интеграция с существующим `DatabaseKeyManager`.

## User Scenarios
1. **Пользователь проходит квиз** – приложение сохраняет результат (баллы, дату) в зашифрованной БД, генерирует цифровой допуск с датой истечения (30 дней). Данные защищены даже при физическом доступе к устройству. Если пользователь проходит квиз до истечения срока предыдущего допуска, дата истечения обновляется.
2. **Сотрудник сканирует QR-код** – приложение валидирует формат, находит в таблице `qr_code_mappings` соответствующие координаты и открывает карту. Если QR-код не найден, показывается сообщение об ошибке с предложением повторить сканирование.
3. **Обновление приложения** – схема БД мигрирует без потери данных, но цифровой допуск сбрасывается (согласно ТЗ). Срок действия допуска не учитывается – он всегда аннулируется при обновлении.
4. **Работа в режиме "в самолёте"** – карта отображается с предзагруженными данными без ошибок, даже при активном Airplane mode.
5. **Отсутствие вопросов в assets** – при запуске квиза, если файл `assets/questions.json` отсутствует или пуст, показывается сообщение об ошибке с предложением повторить попытку позже.

## Functional Requirements
- **FR-01:** Реализовать шифрование Room БД через **SQLCipher** (`net.zetetic:android-database-sqlcipher`). Ключ шифрования генерируется случайным образом (32 байта) и защищается Android Keystore (через `DatabaseKeyManager`). При сбросе ключа в Keystore (например, после сброса устройства) БД автоматически пересоздается с новым ключом, очищая все данные.
- **FR-02:** Создать сущности:
  - `AccessPass` (id, userId, issuedAt, expiryDate, isValid)
  - `QuizResult` (id, userId, score, passed, timestamp)
  - `TrainingVideo` (id, title, fileName, duration)
  - `QrCodeMapping` (objectId, name, latitude, longitude)
  - **Вопросы квиза не хранятся в БД** – они загружаются из `assets/questions.json` при каждом запуске квиза.
- **FR-03:** Разработать DAO:
  - `AccessPassDao` (insert, getValidById, deleteAll)
  - `QuizResultDao` (insert, getByUser)
  - `TrainingVideoDao` (getAll, getById)
  - `QrCodeDao` (getByObjectId, insertOrUpdate, deleteAll)
- **FR-04:** Настроить миграции **1→2→3**:
  - Миграция 1→2: добавление таблицы `TrainingVideo`.
  - Миграция 2→3: добавление поля `isValid` в `AccessPass`.
  - При миграциях сбросить все записи в `AccessPass` (аннулировать допуски).
- **FR-05:** Интегрировать `DatabaseKeyManager` с Room: передать `SupportFactory` в `Room.databaseBuilder().openHelperFactory(factory)`.
- **FR-06:** Реализовать конвертеры для `Date ↔ Long`.
- **FR-07:** Обрабатывать отсутствие файла `assets/questions.json` при запуске квиза: показывать сообщение об ошибке и предлагать повторить попытку позже.
- **FR-08:** Обрабатывать не найденные QR-коды: показывать сообщение об ошибке с предложением повторить сканирование.
- **FR-09:** При обновлении приложения всегда аннулировать допуски (`AccessPass`), независимо от срока их действия.

## Non-Functional Requirements
- **NFR-01:** Время выполнения простых запросов (поиск по `objectId`) – не более 200 мс на типичном устройстве.
- **NFR-02:** Шифрование – AES-256-GCM (SQLCipher с параметрами по умолчанию).
- **NFR-03:** Размер БД ≤ 50 МБ при максимальном количестве записей (до 1000 QR-соответствий, 100 результатов квиза).
- **NFR-04:** Инициализация БД при первом запуске (создание, шифрование, предзагрузка) – не более 3 секунд (с учётом накладных расходов Keystore). Повторные запуски – < 1 сек.
- **NFR-05:** Обработка ошибок при отсутствии вопросов в assets – корректное отображение сообщения без падения приложения.
- **NFR-06:** Обработка ошибок при отсутствии QR-кода в БД – корректное отображение сообщения без падения приложения.

## Boundaries (what is NOT included)
- **Не включена сетевая синхронизация данных** – полный offline.
- **Не включена ротация ключей шифрования** – ключ генерируется один раз при первом запуске и не меняется, кроме случаев сброса устройства.
- **Не включено резервное копирование/восстановление БД**.
- **Не реализована миграция данных, требующая сложного преобразования** – только добавление/удаление таблиц и столбцов.
- **Не требуется поддержка больших объёмов данных >10 000 записей** в одной таблице.
- **Не обрабатывается случай повреждения файла БД** – приложение не восстанавливает поврежденные данные.

## Acceptance Criteria
- [ ] База данных успешно создаётся с шифрованием при первом запуске приложения.
- [ ] Все CRUD-операции через DAO работают корректно (проверено интеграционными тестами Room с in‑memory БД).
- [ ] Миграция с версии 1 на 3 проходит без потери данных (кроме `AccessPass`, который сбрасывается).
- [ ] Запрос по `objectId` в `QrCodeDao` выполняется за <200 мс на реальном устройстве.
- [ ] `DatabaseKeyManager` генерирует и сохраняет ключ в Keystore; при повторных запусках ключ извлекается и успешно открывает БД.
- [ ] При обновлении версии приложения данные в таблицах `QuizResult`, `TrainingVideo`, `QrCodeMapping` сохраняются, а `AccessPass` очищается.
- [ ] При сбросе ключа в Keystore (например, после сброса устройства) БД автоматически пересоздается с новым ключом, очищая все данные.
- [ ] При попытке пройти квиз до истечения срока предыдущего допуска, дата истечения обновляется.
- [ ] При сканировании QR-кода, которого нет в БД, показывается сообщение об ошибке с предложением повторить сканирование.
- [ ] При обновлении приложения срок действия допуска не учитывается – он всегда аннулируется.
- [ ] Карта отображается корректно в режиме "в самолёте" (Airplane mode).
- [ ] При отсутствии файла `assets/questions.json` показывается сообщение об ошибке с предложением повторить попытку позже.
- [ ] **Покрытие бизнес-логики модуля `core-database` не менее 70%** (проверяется через `jacocoTestReport`).
