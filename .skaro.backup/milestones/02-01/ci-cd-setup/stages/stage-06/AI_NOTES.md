## Что было сделано
- Создан GitHub Actions workflow (.github/workflows/ci.yml) для автоматизации сборки, тестирования и проверки качества кода
- Настроены триггеры workflow на push и pull request для веток main и develop
- Интегрированы все проверки из спецификации (QR, root, версия, квиз, шифрование, конвертация PDF)

## Почему такой подход
- Использование GitHub Actions обеспечивает интеграцию с существующей экосистемой GitHub
- Триггеры на push и pull request обеспечивают быструю обратную связь при изменениях кода
- Параллельное выполнение задач (сборка, тесты, линтинг) сокращает время CI
- Загрузка артефактов позволяет легко получить результаты сборки и отчеты
- Тестовые данные созданы в отдельной директории tests/data для удобства поддержки

## Риски и ограничения
- Зависимость от внешних сервисов (GitHub Actions) может быть проблемой в изолированных средах
- Время выполнения CI может увеличиться при добавлении новых проверок
- Тестовые данные могут устареть при изменении форматов QR-кодов или структуры PDF
- Отчеты о покрытии тестами пока являются заглушкой (не реализована реальная интеграция Jacoco)

## Как проверить
1. Запустить локально: `./gradlew assembleDebug testDebugUnitTest detekt ktlintCheck`
2. Запустить все скрипты проверки вручную:
   - `./scripts/validate_qr_format.sh`
   - `./scripts/check_root_detection.sh`
   - `./scripts/check_version_reset.sh`
   - `./scripts/check_quiz_selection.sh`
   - `./scripts/check_database_encryption.sh`
   - `./scripts/check_pdf_conversion.sh`
   - `./scripts/check_navigation.sh`
3. Создать PR и проверить, что workflow проходит все этапы
4. Проверить артефакты сборки в разделе Actions GitHub
